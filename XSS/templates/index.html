üîç How Reflected XSS Attacks Work

Attack Flow:

Attacker crafts a malicious URL with JavaScript code embedded in a parameter
Attacker tricks user into clicking the link (via email, chat, forum, etc.)
User's browser sends the request to the vulnerable server with the malicious code
Server reflects the code back in the HTML response WITHOUT sanitizing it
Browser executes the JavaScript - attacker gains access to user's data/session

Example URL Attack:

http://localhost:5000/?q=<img src=x onerror="alert('Your data stolen!')">

What Can Attackers Do?

  • ‚úì Steal session cookies and authentication tokens
  • ‚úì Redirect users to phishing websites
  • ‚úì Capture keystrokes and form data
  • ‚úì Modify page content to trick users
  • ‚úì Download malware or execute drive-by downloads
  • ‚úì Spread worms through user accounts

Key Difference: Reflected vs Stored XSS

Reflected XSS Stored XSS
In URL/parameter, reflected back immediately Saved in database, executed for all visitors
Attacker must trick user into clicking link No user interaction needed - passive infection
Less dangerous (single user affected) More dangerous (ALL users affected)
